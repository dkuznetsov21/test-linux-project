### CSS_APPROACH: {{CSS_APPROACH}}

| Approach | Class pattern | Install | Compatible with |
|---|---|---|---|
| tailwind | `bg-blue-500 text-white px-4` | tailwindcss, postcss, autoprefixer | all frameworks |
| css-modules | `.module.css` -> hashed classes (`header_a3f2`) | built-in (no extra deps) | react, vue, svelte |

## A2. Image Fallback System (MANDATORY)

Create a reusable component in the framework's syntax:
- React: src/components/ImageWithFallback.tsx
- Vue: src/components/ImageWithFallback.vue
- Svelte: src/components/ImageWithFallback.svelte
- Astro: src/components/ImageWithFallback.astro

Props interface (adapt to framework syntax):
```typescript
interface ImageWithFallbackProps {
  src: string;
  alt: string;
  className?: string;
  fallbackType?: 'gradient' | 'pattern' | 'icon';
  aspectRatio?: string;
}
```

RULES:
1. Use Unsplash source URLs: https://images.unsplash.com/photo-{ID}?w=800&h=600&fit=crop
2. EVERY <img> in the project MUST use ImageWithFallback — no raw <img> tags.
3. On image load error, the component MUST:
   - Hide the broken image
   - Show a styled CSS gradient placeholder using site accent colors
   - OR show an inline SVG geometric pattern
   - NEVER show a broken image icon
4. Hero section backgrounds: use CSS gradients as the base layer, with optional image overlay via background-image. The section must look complete even if the image fails to load.
5. Avatar images: use https://ui-avatars.com/api/?name=First+Last&background=random as primary source, with SVG initials circle as fallback.
6. Centralize all image URLs in src/data/images.ts. Every URL unique (no repetition). Alt text in {{LOCALE}}.

## A5. Copywriting Rules (Ad Platform Compliant)

### FORBIDDEN (triggers ad rejection)

NEVER use:
- "Guaranteed results" / "We guarantee..."
- Income/earnings promises ("Make $X", "Earn $X")
- "Secrets of..." / "What they don't tell you..."
- Fake scarcity ("Last spot!", "Only X left!")
- Fake urgency ("Today only!", "Offer ends in...")
- "Effortless" / "Easy money" / "Quick results"
- "Exclusive method" / "Proven system"
- "100% effective" / absolute claims
- Countdown timers for fake deadlines
- Testimonials with specific income/result numbers
- Before/after with unrealistic transformations
- "Sign up now before it's too late"
- Fake celebrity endorsements
- "Thousands already succeeded"
- "Change your life" / life-changing promises
- Superlatives without proof ("Best", "Leading", "#1")
- Pressure tactics / FOMO manipulation

### TESTIMONIAL RULES

Each testimonial MUST:
- Focus on EXPERIENCE, not RESULTS
- Sound human (not all glowing, some constructive notes OK)

TESTIMONIAL FORMAT POOL (use 1+ formats per site, mix on same page):
a) Full: name + role + company
b) Minimal: first name + city only
c) Platform-style: name + rating stars + verified badge
d) Anonymous: quote + "Client in [industry]"
e) Mixed: 2+ formats on same page

TESTIMONIAL VOICE VARIATION:
- LENGTH: Mix 1-sentence, 2-sentence, 3-sentence on same page. NEVER all same length.
- TONE: Some formal, some casual, some technical/specific
- STRUCTURE:
  a) Positive + minor criticism at end
  b) Problem -> solution story
  c) Descriptive experience
  d) Mentions specific feature/person
  e) One short punchy sentence
- At least 1 per site with mild reservation
- NEVER have all testimonials same length or structure

GOOD examples (write NEW text each time, NEVER copy these verbatim):
- "The [specific module name] gave me a repeatable process for [specific task]. I still use the template from week 3." — [Name], [Role]
- "Clear material, though the [specific topic] section felt compressed. Mentor notes on my code were the standout." — [Name]
- "We ran the [specific exercise] in our team the next sprint. Solid framework." — [Name], [Company]

BAD (NEVER write these):
- "Thanks to this I doubled my [metric] in 3 months! Best decision ever, 100% recommended!!!"

### If CSS_APPROACH = "plain-css":

LAYOUT_ARCHETYPE: {{LAYOUT_ARCHETYPE}}

The archetype defines the overall page structure, spacing philosophy, and navigation approach.
All other design decisions (colors, type, decorative elements) come from Layer C theme.

### corporate
- Top horizontal nav bar (fixed/sticky)
- Banner hero (full-width background)
- Content sections in centered container (max-w-6xl to max-w-7xl, ~1280px)
- 3-column feature grids
- Multi-column footer (3-4 columns)
- Section spacing: py-16 to py-24

### magazine
- Multi-column editorial grid (2-3 columns with varying widths, CSS grid)
- Pull quotes and large typography callouts
- Text-heavy layout, images used editorially (not decoratively)
- Asymmetric layouts — content NOT always centered
- Thin hairline dividers between sections
- Narrow content column: max-w-prose (~65ch) for text, wider for media
- Section spacing: py-20 to py-32

### minimal
- Maximum whitespace (py-32 to py-48 between sections)
- Very few elements per viewport height
- Full-bleed images, large scale
- Centered narrow text column (max-w-2xl, ~672px)
- Minimal footer (single row: brand + legal + copyright)
- Minimal nav (3-4 items max, rest in footer)
- Section spacing: py-32 to py-48

### landing

CONTENT UNIQUENESS RULES (MANDATORY)
================================================================

These rules exist because generated sites tend to produce identical copy. Each rule addresses a
specific pattern that has been observed repeating across multiple generations.

### Cookie consent text
NEVER use "[BRAND] uses cookies for functionality and analytics."
NEVER use "We use cookies for [X]. See [Cookie Policy]"
Write a genuinely NEW sentence each time. Pool for inspiration:
- "We store small data files to keep the site working. Details in our [link]."
- "This website sets essential cookies. Analytics cookies require your opt-in. [link]"
- "Cookies help deliver our content. Choose your preferences below."
- "We respect your privacy — only essential cookies are set by default. [link]"
- "Small text files help us improve this site. You choose what to allow."
But do NOT copy these examples verbatim — write a new variation each time.

### Legal link names and order
NEVER use this exact set in this order: Privacy Notice, Cookie Settings, Terms of Use, Refund & Cancellation.
Each site MUST use different names AND different order from the A7 pools.
Shuffle BOTH names AND order based on {{SITE_VARIANT_SEED}}.

### Testimonials
NEVER write testimonials containing any of these phrases:
- "helped me structure my approach better"
- "practical exercises were the most valuable part"
- "Would have liked more time on [topic]"
- "Best decision ever"
These have appeared verbatim across multiple sites.
Each testimonial MUST reference SPECIFIC {{ITEM_TYPE_PLURAL}} names or features from THIS site.

### Hero headline
NEVER default to question format for all sites ("Need X?", "Ready to Y?", "Want to Z?").
Use hero content structure variants from B4 pool. Spread across:
- Statement: "Backend pipelines built with feedback loops"
- Stat-driven: "400+ engineers trained since 2019"
- Testimonial-driven: opening with a client quote
- Narrative: two paragraphs, no big H1 question
- Minimal: short phrase, no subtitle
- List-driven: H1 + 3 bullet outcomes

### CTA section
NEVER use "Next cohort: [Month Year]" + "Limited to 25 participants per cohort for quality".
Vary CTA approach from this pool:
- Newsletter/mailing list signup
- Download syllabus / brochure / PDF
- Book a demo / consultation call
- Apply for next intake
- Join waitlist with email
- "Enquire about [specific program]"
Participant caps: vary or omit — use 15, 20, 30, 12, or no cap mentioned.

### Stats section
NEVER use exactly 4 metrics with this pattern: [item count] + satisfaction + max participants + mentors.
Vary metric count: 3, 5, or 6 (NOT always 4).
Vary metrics from: years active, countries served, projects completed, hours of content,
completion rate, alumni employed, partner organizations, languages, cities, client companies.
Vary format: some as plain numbers, some as percentages, some with units.

### Section headings
NEVER use all of these defaults together: "What you get", "How it works", "What participants say", "Common questions".
Vary heading style: some descriptive, some action-oriented, some single-word, some with dashes/numbers.
Examples: "Inclusions", "The flow", "From our cohorts", "FAQ", "01 — Process", "Three things we do differently".

### Page <title>
NEVER use "BRAND — Topic Courses" on every site. See TITLE FORMAT POOL in B8.

DESIGN VARIATION GUIDELINES
================================================================

## Visual Cliches to AVOID
- Perfect 3-column grids everywhere
- Identical card heights/padding
- Exact 50/50 hero splits
- Same icon family in every section
- Everything perfectly centered
- Same animation on all elements
- Stock photo aesthetic (too polished)

## Structural Cliches to AVOID
- "Hero -> Features -> How it works -> Testimonials -> CTA" exact order on every site
- Every section with exactly 3 items
- Perfectly symmetrical everything
- Identical section structures repeated
- Same page set every generation

- "We are leaders in..."
- "Our team of experts..."
- "Comprehensive solutions"
- "Take your X to the next level"
- "In today's world..."
- "Digital transformation..."
- "Synergy" / "Leverage" / "Optimize"
- "World-class" / "Best-in-class"
- "Passionate about..."

- Vary section order by page purpose and {{SITE_VARIANT_SEED}}
- Mix 2, 4, 5 item sections (avoid always-3)
- Use asymmetric layouts
- Write specific copy for {{INDUSTRY}}, not generic filler
- Include minor imperfections (one testimonial shorter, etc.)
- Use unexpected section combinations
- Break visual rhythm intentionally
